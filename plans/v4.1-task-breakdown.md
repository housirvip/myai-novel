# AI 自动写小说工具 v4.1 任务拆解

## 1. 这份文档的定位

- 这份文档是 `v4` 之后的收口与强化版本
- 它不再重复建设 `v4` 已经具备的卷级真源、卷级 CLI 基础骨架与命令拆分结构
- 它聚焦当前最有价值、最影响系统稳定度与可用度的剩余工作：
  - 把卷级能力从“可调用”推进到“可验证”
  - 把卷级诊断从“启发式摘要”推进到“更可信的规则化诊断”
  - 把 generation / rewrite 从“已接入卷级约束”推进到“能围绕长线问题定向工作”
  - 把 regression 从“样本清单”推进到“真实执行器”
- 这份文档的目标，是回答一个更实际的问题：

> 在 `v4` 已基本打通多章导演主链之后，下一阶段最应该补哪几块，才能让系统真正进入“稳定可用的创作工作流”状态？

---

## 2. 当前基线判断

基于当前代码状态，`v4` 已经具备以下基础：

- 卷级真源已具备：
  - `VolumePlan`
  - `StoryThread`
  - `StoryThreadProgress`
  - `EndingReadiness`
- planning / review / approve 主链已经消费卷级语义
- generation / rewrite 已接入卷级导演约束
- `chapter / workflow / state / doctor / snapshot / regression` 命令域已经按 `register + subcommand + services + printers` 模式基本收敛
- 卷级 CLI 已有第一批可用命令：
  - `plan volume-window <chapterId>`
  - `plan volume-show <volumeId>`
  - `plan mission-show <chapterId>`
  - `review volume <volumeId>`
  - `state threads [volumeId]`
  - `state ending`
  - `state volume-plan <volumeId>`
  - `state volume <volumeId>`
  - `doctor volume <volumeId>`
  - `snapshot volume <volumeId>`
  - `regression volume <volumeId>`
- `v4` 样本口径文档已具备：`plans/v4-regression-cases.md`

但还存在明显缺口：

- regression 还是骨架，尚未成为真正的回归执行器
- volume doctor 的主线停滞 / 群像失衡 / mission 断链仍偏启发式
- rewrite 虽然已接入卷级约束，但还没有真正产品化成可切换的卷级修复策略
- generation 虽然看到了卷级导演信息，但“不可抢跑收束 / 不可拖延主线”的约束还没有足够明确的执行口径
- 注释强化还未彻底补齐到所有核心主链决策点

因此，`v4.1` 不再以“铺新结构”为主，而以“让现有结构变得可信、可回归、可验收”为主。

---

## 3. v4.1 总目标

`v4.1` 的总目标定义为：

> 把 `v4` 已经搭好的多章导演系统，从“结构上成立”推进到“运行上可信、诊断上明确、回归上可执行”。

更具体地说，`v4.1` 要解决四个问题：

1. 系统能否真正验证卷级能力没有跑偏？
2. 系统能否对卷级问题给出更可信的诊断，而不只是模糊提示？
3. generation / rewrite 是否真的受卷级 mission / thread / ending 约束，而不是只在 prompt 里提到？
4. 当前命令体系、样本体系、诊断体系能否形成稳定的验收闭环？

---

## 4. v4.1 执行策略

`v4.1` 不建议再大范围扩张领域模型，而建议按下面顺序推进：

1. 先把 regression 执行器做出来
2. 再把 volume doctor 规则化增强
3. 再把 rewrite 卷级策略显式化
4. 再把 generation 的卷级约束进一步收紧
5. 最后补注释、样本、验收口径与文档收尾

原因：

- 没有 regression 执行器，后续每一次增强都缺少稳定验证手段
- 没有更可信的 volume doctor，卷级 CLI 只能“能看”，不能真正辅助判断风险
- rewrite 是当前最像“约束已接入但能力未产品化”的环节，继续做深收益很高
- generation 的 prompt 已具备卷级字段，但约束的执行口径还可更明确
- 注释与文档应在主链稳定后再收尾固化，性价比更高

## 6. 里程碑拆解

## V4.1-A：回归执行器落地

### 目标

把当前 `regression` 从“样本命名与输出骨架”推进为“可执行的手工/半自动回归驱动器”。

### 模块任务

#### A. `src/cli/commands/regression/run.ts`

补强：

- 根据 `caseName` 分发到不同回归 case 执行器
- 至少支持第一批 `v4` case：
  - `volume-plan-smoke`
  - `mission-carry-smoke`
  - `thread-progression-smoke`
  - `ending-readiness-smoke`
  - `volume-doctor-smoke`
- 输出统一的 case 执行结果结构：
  - `caseName`
  - `status`
  - `steps`
  - `artifacts`
  - `summary`

#### B. `src/cli/commands/regression/volume.ts`

补强：

- 不再只展示“保留的 case 列表”
- 支持按 `volumeId` 运行卷级 case 套件
- 输出卷级回归摘要

#### C. `src/cli/commands/regression/printers.ts`

补强：

- 增加回归执行结果打印格式
- 增加失败 case 汇总格式
- 增加 artifact 输出格式

#### D. `plans/v4-regression-cases.md`

补强：

- 为每个 case 增加：
  - 预期输入前置条件
  - 预期输出检查点
  - 失败后重点查看命令
  - 建议保留 artifact

### 完成定义

- `regression run <caseName>` 不再只是 skeleton
- `regression volume <volumeId>` 能运行卷级回归集合
- 卷级能力有稳定回归入口

---

## V4.1-B：volume doctor 规则化增强

### 目标

让 `doctor volume` 从“卷级摘要”升级成“更可信的卷级创作风险诊断器”。

### 模块任务

#### A. `src/cli/commands/doctor/volume-services.ts`

继续补强：

- 把“主线停滞”从简单数量信号推进为更明确的规则：
  - 高优先级线程长期停留在低阶段
  - 多章未形成 progress 更新
- 把“mission 断链”做得更严格：
  - mission 指向的 chapter 不存在
  - mission 与 volume window 不一致
  - 当前卷章节没有被 mission 覆盖
- 把“群像失衡”从近似信号推进为更可解释的规则：
  - 角色线长期未推进
  - 关系线被反复跳过
  - 支线承接要求长期未兑现
- 把“终局风险”输出做得更结构化：
  - `closureGapCount`
  - `neglectedThreadCount`
  - `endingTargetMatches`
  - `pendingPayoffPressure`

#### B. `src/cli/commands/doctor/volume-printers.ts`

补强：

- 输出不只是一段 diagnostics JSON
- 拆成：
  - 总体风险摘要
  - mission 风险
  - thread 风险
  - ending 风险
  - chapter 风险

#### C. `src/cli/commands/doctor/volume.ts`

补强：

- 保持入口不变
- 支持后续可能的 `--json` 或 `--strict` 选项预留位

### 完成定义

- `doctor volume <volumeId>` 可以更清晰地指出：
  - 哪条线程停滞
  - 哪个 mission 断链
  - 哪些收束缺口最危险
  - 哪类群像问题正在累积

---

## V4.1-C：rewrite 卷级策略显式化

### 目标

让 rewrite 不只是“知道卷级约束”，而是能围绕卷级问题切换更明确的修复策略。

### 模块任务

#### A. `src/core/rewrite/service.ts`

补强：

- 为当前策略层显式补出三类卷级修复焦点：
  - `thread-focus`
  - `closure-focus`
  - `ensemble-balance`
- 在 `resolveRewriteStrategyProfile()` 中加入基于 review / goals / volume signals 的切换逻辑
- 在 `buildRewriteQualityTarget()` 中加入卷级问题焦点
- 在 `summarizeRewriteContext()` 中强化：
  - mission
  - thread focus
  - carry tasks
  - ending drive
  - ensemble focus

#### B. `src/core/review/service.ts`

补强：

- 为 rewrite 策略建议提供更明确的来源
- 把卷级问题与建议策略映射得更直接

### 完成定义

- rewrite 能更明确地区分：
  - 主线推进不足型修复
  - 收束准备不足型修复
  - 群像失衡型修复
- rewrite 不再只是“通用修文”，而是“卷级导演约束下的定向修复”

---

## V4.1-D：generation 约束口径收紧

### 目标

让 generation 更明确地遵守“不可抢跑收束 / 不可拖延主线 / 不可继续放大群像失衡”。

### 模块任务

#### A. `src/core/generation/service.ts`

补强：

- 在 prompt 中明确“当前章不是终局章时，不可过早回收关键终局项”
- 在 prompt 中明确“高优先级 thread 不可继续拖延”
- 在 fallback 生成中补出更明确的卷级约束段落

#### B. `src/core/context/writing-context-builder.ts`

补强：

- 更明确表达当前章在卷窗口中的位置
- 为 generation 提供更直接的 mission / thread priority / ending readiness 视图

### 完成定义

- generation 对卷级 mission 的执行更稳定
- generation 对“过早收束 / 拖延主线”的违约概率降低

## 7. 推荐执行顺序

建议后续严格按下面顺序推进：

1. `V4.1-A`：先做 regression 执行器
2. `V4.1-B`：再做 volume doctor 规则化增强
3. `V4.1-C`：再做 rewrite 卷级策略显式化
4. `V4.1-D`：再做 generation 约束口径收紧
5. `V4.1-E`：最后做注释、文档与收尾验收

原因：

- 没有 regression 执行器，后续增强缺少稳定验证手段
- doctor 更可信后，才能更准确评估卷级问题
- rewrite 是当前最明显的“已接入但未产品化”环节
- generation 收紧应放在已有回归与诊断之后，以便及时验证
- 注释与文档适合在主链趋稳后统一补齐
