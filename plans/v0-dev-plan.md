# AI 自动写小说工具 v0 开发计划

## 1. v0 定位

- 目标：用最小可实现方案跑通一次完整写作主链路
- 形态：本地单用户 CLI 工具
- 存储：仅支持 SQLite
- 模型：仅接入一个 LLM Provider
- 原则：先验证流程闭环，不追求复杂状态系统与高级回滚能力

一句话定义：

> v0 不是“完整小说操作系统”，而是“能从设定录入到章节终稿确认跑通的最小写作闭环”。

## 2. v0 成功标准

满足以下条件即可视为 v0 完成：

1. 能初始化一个小说项目
2. 能录入基础设定：书籍、大纲、卷、章、角色、地点、势力、钩子
3. 能为指定章节生成章节计划包
4. 能基于章节计划包生成章节草稿
5. 能对草稿执行自审并输出审查报告
6. 能对当前章节执行定向重写
7. 能执行 `approve`，生成终稿并提交核心状态更新
8. 能将章节终稿输出到 `completed-chapters/`
9. 能在重启程序后重新读取 SQLite 中的数据继续工作

## 3. v0 明确不做

- MySQL 支持
- 多模型切换
- 跨章节回滚与整链重放
- 完整物品全局审计闭环
- 高级长期记忆去重和召回排序
- 管理型 debug/admin 命令
- 图形界面
- 自动发布平台对接

## 4. v0 交付范围

### 4.1 必须交付

- `novel init`
- `novel outline set`
- `novel volume add`
- `novel chapter add`
- `novel character add`
- `novel location add`
- `novel faction add`
- `novel hook add`
- `novel plan chapter <id>`
- `novel write next`
- `novel review chapter <id>`
- `novel chapter rewrite <id>`
- `novel chapter approve <id>`

### 4.2 最小状态更新范围

`approve` 后只要求更新以下内容：

- `Chapter.status`
- 当前推进章节
- 主角当前位置
- 最近关键事件
- 活跃钩子状态
- 短期记忆
- 轻量长期记忆候选

### 4.3 允许人工介入的点

- 章节计划包允许人工确认后再写作
- 审查报告只负责指出问题，不要求 100% 自动修复
- 重写策略由用户指定目标
- 状态更新允许先由 LLM 提取，再由规则层做基础校验

## 5. 技术落地策略

### 5.1 工程结构

建议首版目录：

```text
src/
  cli/
  core/
    planning/
    generation/
    review/
    rewrite/
    state/
    hooks/
    memory/
    context/
  infra/
    db/
    llm/
    repository/
    audit/
  shared/
    types/
    utils/
```

### 5.2 数据策略

- SQLite 作为唯一数据库实现
- 数据库是真相源
- 文件系统只保存终稿和运行日志
- 章节过程记录保留，但 v0 不做可回放重建

### 5.3 类型策略

- `Entity`：数据库真源
- `View`：上下文和查询结果
- `Contract`：工作流输入输出
- v0 代码中不要混用这三类类型

## 6. 里程碑拆分

## M1：工程骨架可运行

目标：把 CLI、SQLite、类型和基础目录跑起来。

交付物：

- TypeScript CLI 可执行入口
- SQLite 连接与迁移机制
- 基础目录初始化逻辑
- 基础共享类型文件
- 日志与错误处理骨架

完成标准：

- 执行 `novel init` 能创建项目目录与 SQLite 文件
- 再次执行不会破坏已有项目
- 基础迁移可以成功执行

---

## M2：基础设定录入可用

目标：把静态设定写入数据库，并可以查询读取。

交付物：

- `Book`、`Outline`、`Volume`、`Chapter` 仓储
- `Character`、`Location`、`Faction`、`Hook` 仓储
- 对应 CLI 录入命令
- 基础参数校验

完成标准：

- 能完成一部书的基础设定录入
- 所有录入命令都能幂等或给出明确错误
- 数据重启后可重新读取

---

## M3：章节计划链路打通

目标：先让“规划”成为正文生成前的明确中间层。

交付物：

- `PlanningContext` 组装器
- `ChapterPlan` 生成器
- 计划包落库
- 基础计划校验器

完成标准：

- 对指定章节可生成计划包
- 计划包包含场景、关键事件、角色/地点/势力/钩子约束
- 计划包可被后续写作阶段直接使用

---

## M4：正文生成与审查可用

目标：基于计划包生成草稿，并形成审查反馈。

交付物：

- `WritingContext` 组装器
- 草稿生成器
- `ReviewReport` 生成器
- 审查结果落库

完成标准：

- `write next` 能生成草稿
- `review chapter` 能输出一致性、字数、节奏和钩子问题
- 草稿和审查记录可追溯

---

## M5：重写与确认发布打通

目标：让章节真正进入“可修改、可确认、可落盘”的状态。

交付物：

- `RewriteRequest` 处理逻辑
- 候选版本管理
- `approve` 事务提交逻辑
- 终稿导出到 `completed-chapters/`

完成标准：

- 可以对当前章节发起定向重写
- 重写结果不会直接污染主线状态
- `approve` 后会生成终稿并更新章节状态为 `finalized`

---

## M6：最小状态更新闭环

目标：让章节确认后，系统能留下足够的连续写作状态。

交付物：

- 章节确认后的状态更新服务
- 钩子状态更新
- 短期记忆更新
- 轻量长期记忆沉淀
- 审计日志记录

完成标准：

- `approve` 后可更新当前章节指针
- 主角位置和最近事件可更新
- 活跃钩子与已回收钩子可更新
- 下一章可读取到这些状态继续生成

## 7. 推荐执行顺序

### 第 1 周：先打骨架

1. 初始化 CLI 与项目目录
2. 建立 SQLite 与迁移系统
3. 建立共享类型、仓储接口与错误模型
4. 完成设定录入命令与基础查询

### 第 2 周：打通主链路

1. 实现章节计划包生成
2. 实现基于计划包的正文生成
3. 实现审查报告
4. 实现定向重写
5. 实现 `approve` 与终稿输出

### 第 3 周：补状态闭环

1. 实现最小状态更新
2. 实现钩子更新
3. 实现短期记忆与轻量长期记忆
4. 补审计日志与流程稳定性

如果你想更保守，v0 也可以压缩为两阶段：

- v0.1：只做 `init -> set -> plan -> write -> review`
- v0.2：补 `rewrite -> approve -> state update`

## 9. 每阶段验收口径

### M1 验收

- 新项目可初始化
- SQLite 文件可创建
- 基础迁移可重复执行

### M2 验收

- 所有基础设定可通过 CLI 录入
- 录入后可重新读取

### M3 验收

- 指定章节可生成计划包
- 计划包落库成功

### M4 验收

- 能生成草稿
- 能输出审查报告

### M5 验收

- 能定向重写当前章节
- 能确认终稿并导出

### M6 验收

- 能更新章节推进状态
- 能更新钩子和记忆
- 下一章能利用这些状态继续工作

## 10. 风险控制

### 风险 1：上下文过重导致提示词失控

应对：

- 规划和写作分成两个上下文
- 长期记忆只召回高优先级条目
- 先保证可写，再逐步扩充上下文密度

### 风险 2：状态更新不稳定

应对：

- v0 只更新最小必要状态
- 先结构化输出，再做规则校验
- `approve` 前不写主线状态

### 风险 3：命令过多导致开发分散

应对：

- 所有设计围绕主链路命令
- 低频调试能力延后

## 11. 推荐启动方式

如果你现在就开始开发，我建议按下面顺序开工：

1. 先做 `novel init`
2. 再做 SQLite + migration
3. 再做 `outline / volume / chapter / character` 的录入命令
4. 然后做 `plan chapter`
5. 再做 `write next`
6. 再做 `review chapter`
7. 最后做 `rewrite` 和 `approve`

一句话原因：

> `plan -> write -> review -> approve` 是 v0 的生命线，其他能力都应该为它服务。
