# AI 自动写小说工具 v5.1 任务拆解

## 1. 这份文档的定位

- 这份文档不是重做 `v5`
- 它的目标，是把 `v5` 已经搭好的 **多 provider LLM + SQLite/MySQL 双后端基础设施**，从“主链已打通”推进到“可验收、可观测、可长期维护”
- 它只聚焦当前 `v5` 剩余的那部分高价值收口工作，不再引入新的大范围产品方向

换句话说，`v5.1` 要解决的不是“还能不能接上 MySQL / 第二个 provider”，而是：

> 这些能力是否已经足够稳定、足够透明、足够容易被用户启用，并且在后续迭代里不容易再退化？

---

## 2. 当前基线判断

基于当前代码状态，可以把 `v5` 的现状概括为：

### 已基本完成的部分

- 已建立 LLM provider 抽象与工厂分发
- 已存在至少两个 provider 形态：
  - `OpenAI`
  - `OpenAI-compatible`
- `planning / generation / review / rewrite` 主链已经接入统一 LLM 工厂
- 数据底座已经从强绑定 SQLite 向统一 DB client 抽象演进
- 已补 MySQL adapter、driver 选择与项目数据库打开流程
- repository 已大面积补齐 async 访问面
- `state / doctor / workflow / regression` 的关键只读链路已收口到 async MySQL 路径
- `doctor` / `regression` 已开始承载基础设施层检查职责

### 尚未完全收口的部分

- LLM 输出元数据虽有基础，但还没有形成统一、稳定、贯穿主链的产品口径
- fallback / retry / timeout 的能力边界还不够显式，也未完全沉淀成统一 gateway 行为
- MySQL 虽已进入“可运行”阶段，但离“完整验收基线”还差系统性回归与命令级覆盖确认
- `init / migrate / doctor / regression / 文档` 对新基础设施能力的说明和验收口径还不够一致
- `.env.example`、`COMMAND_GUIDE.md`、`plans/` 中还缺少真正对用户友好的启用说明与验收闭环

因此，`v5.1` 的重点不再是“继续铺底层接口”，而是：

- 补完可观测性
- 补完回归面
- 补完文档与启用说明
- 补完 MySQL / provider 切换场景下的产品化收口

---

## 3. v5.1 总目标

`v5.1` 的总目标定义为：

> 把 `v5` 已经打通的基础设施能力，从“工程上可用”推进到“产品上可信、切换上可控、回归上可守”。

更具体地说，`v5.1` 需要回答四个问题：

1. LLM provider 的选择、路由、失败与 fallback 是否足够透明？
2. MySQL 项目是否已经具备“初始化—迁移—主链—状态命令—诊断—回归”的完整闭环？
3. 用户是否能通过文档和示例配置，明确知道如何启用 provider 与数据库后端？
4. 后续再改基础设施时，是否已有足够的 regression / doctor 保护面防止退化？

---

## 4. v5.1 执行策略

建议按下面顺序推进：

1. 先补 **LLM 可观测性与失败治理**
2. 再补 **MySQL 全链路验收与缺口修复**
3. 再补 **doctor / regression 的基础设施保护面**
4. 最后补 **文档、示例配置与验收口径**

原因：

- 如果没有 LLM 元数据与失败分类，后续 provider 问题难以定位
- 如果没有 MySQL 命令级验收，当前“已接通”很容易在边角命令继续失效
- 如果没有回归保护面，后续小重构很容易把多 provider / MySQL 路径再次打断
- 文档和示例配置应基于最终实现口径收尾，否则会反复返工

## 6. 里程碑拆解

## V5.1-A：LLM 可观测性与失败治理收口

### 目标

让 LLM 能力从“能切 provider / model”升级成“能解释当前用了谁、为什么失败、是否发生 fallback”。

### 模块任务

#### A. `src/shared/types/domain.ts`

补强统一 LLM 元数据结构：

- `provider`
- `model`
- `latencyMs`
- `retryCount`
- `fallbackSource`
- `finishReason`
- `rawUsage` / token usage 摘要
- `requestId` / traceId（如可获得）

#### B. `src/infra/llm/factory.ts`

补强 gateway 层职责：

- 明确 provider 选择来源
- 统一 timeout / retry 策略入口
- 为 provider 不可用、配置错误、响应结构异常提供明确错误分类
- 为 fallback 记录来源与最终命中 provider

#### C. `src/infra/llm/openai-adapter.ts` / `src/infra/llm/openai-compatible-adapter.ts`

补强：

- metadata 回填
- 错误标准化
- 超时/重试后错误口径统一
- 返回结构一致性校验

#### D. `src/core/planning/service.ts` 等四条主链

补强：

- 把 LLM metadata 写入日志/结果对象/调试输出链路
- 为结构化输出失败场景补统一降级说明
- 让阶段级模型选择在运行结果里可见，而不是只存在于配置

### 完成定义

- 每次 LLM 调用都能追踪 provider / model / fallback 信息
- provider 配置错误和响应错误可快速区分
- planning / generation / review / rewrite 四条链路都能输出统一 metadata

---

## V5.1-B：MySQL 全链路验收与补洞

### 目标

把 MySQL 从“主链基本可跑”推进到“项目级闭环可验收”。

### 模块任务

#### A. `src/infra/db/migrate.ts`

补强：

- 明确 SQLite / MySQL 方言边界
- 为 migration 执行输出更清晰的 backend 信息
- 对 MySQL 初始化失败、连接失败、schema 不兼容给出更清晰错误口径

#### B. `src/cli/context.ts`

补强：

- 打开数据库时输出/记录当前 backend 信息
- 对项目缺配置、配置不合法、连接失败场景做更明确分层

#### C. `src/cli/commands/project-commands.ts`

补强：

- `init` 流程对 SQLite / MySQL 的初始化分支做完整验收
- 明确新项目选择后端的方式
- 对旧 SQLite 项目保持无迁移可运行的兼容口径

#### D. `src/infra/repository/` + 命令域

继续排查并补齐：

- 剩余未完全 async 化的边角仓储
- 仍误走同步接口的 MySQL 路径
- SQLite/MySQL SQL 方言差异点

建议重点回归：

- `project`
- `chapter`
- `workflow`
- `state`
- `doctor`
- `snapshot`
- `regression`

### 完成定义

- MySQL 项目可完成：初始化、迁移、主链、核心状态查看、doctor、regression
- 常用命令不再因同步仓储/方言差异在 MySQL 下失效
- SQLite 兼容性不回退

## 9. 边界

`v5.1` 仍然**不包含**：

- 自动数据迁移工具
- 多实例并发治理
- 在线服务化部署
- 分布式任务队列
- SaaS 多租户隔离

`v5.1` 只聚焦：

- `v5` 基础设施能力的收口
- provider / backend 切换的可观测性
- MySQL 闭环验收
- doctor / regression / 文档 / 配置示例 的一致性建设

如果说 `v5` 的任务是“把多 provider LLM 与 SQLite/MySQL 双后端接进系统”，那么 `v5.1` 的任务就是：

> 把这些能力真正变成可验证、可诊断、可启用、可维护的长期基础设施。
