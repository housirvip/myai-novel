# AI 小说工具 V1 方案

## 1. 目标

打造一个基于 TypeScript + Node.js 的本地 CLI 小说创作工具，核心能力包括：

- 管理一本书的长期设定与章节生产流程
- 抽象多种 LLM 提供方，支持 `mock`、`openai`、`anthropic`、`custom`
- 当前基于 SQLite 落地，后续可平滑切换到 MySQL
- 用结构化数据管理世界观、人物、势力、钩子、章节等信息
- 通过 CLI 驱动完整章节流水线：`plan -> draft -> review -> repair -> approve`

## 2. 技术选型

### 2.1 技术栈

- 语言：TypeScript
- 运行时：Node.js 20+
- CLI：`commander`
- 配置：`dotenv`
- 数据库：SQLite
- SQLite 驱动：`better-sqlite3`
- SQL/数据库抽象：`kysely`
- 校验：`zod`
- 日志：`pino`

### 2.2 选型理由

- `commander` 适合做清晰的子命令风格 CLI
- `dotenv` 便于管理不同 LLM 的 API Key
- `kysely` 可同时支持 SQLite / MySQL，适合提前预留数据库方言切换口
- `better-sqlite3` 同步 API 简洁，适合本地单机 CLI 工具
- `zod` 可用于 CLI 参数、配置和 LLM 输出结构校验

## 3. V1 范围

### 3.1 包含

- 单机 CLI 工具
- 单用户、本地项目目录运行
- 结构化存储一本书及其设定
- 章节规划、生成、审阅、修复、批准
- 基于关键词和实体召回上下文
- `approve` 时将章节最终事实同步回设定库

### 3.2 暂不包含

- Web UI
- 多人协作
- 向量数据库
- 自动长时任务调度
- 精细权限系统

## 5. 配置设计

`.env` 建议包含：

```dotenv
NODE_ENV=development
LOG_LEVEL=info
LOG_FORMAT=pretty
LOG_DIR=./logs
LOG_LLM_CONTENT_ENABLED=false
LOG_LLM_CONTENT_MAX_CHARS=4000

DB_CLIENT=sqlite
DB_SQLITE_PATH=./data/novel.db

LLM_PROVIDER=mock

OPENAI_API_KEY=
OPENAI_BASE_URL=
OPENAI_MODEL=

ANTHROPIC_API_KEY=
ANTHROPIC_MODEL=

CUSTOM_LLM_BASE_URL=
CUSTOM_LLM_API_KEY=
CUSTOM_LLM_MODEL=
```

约束说明：

- `DB_CLIENT` 先支持 `sqlite`，后续可扩展为 `mysql`
- `LLM_PROVIDER` 支持 `mock | openai | anthropic | custom`
- 每个 provider 的模型名和地址配置彼此独立
- `LOG_FORMAT` 支持 `pretty | json`
- `LOG_DIR` 用于落地日志文件
- `LOG_LLM_CONTENT_ENABLED` 控制是否记录 AI 输入输出正文，默认 `false`
- `LOG_LLM_CONTENT_MAX_CHARS` 控制记录的最大字符数，避免日志爆炸

### 5.1 日志配置建议

V1 建议同时支持控制台日志和文件日志：

- 控制台日志
  - 本地开发默认 `pretty`
- 文件日志
  - 默认写入 `./logs/app-YYYY-MM-DD.log`
- 错误日志
  - 可额外写入 `./logs/error-YYYY-MM-DD.log`

所有 CLI 命令都应生成一个 `traceId`，贯穿：

- CLI 命令入口
- service / workflow
- repository
- LLM provider 调用

AI 内容日志建议采用“双保险”：

- 默认关闭
- 需要排查时通过环境变量或命令参数显式开启

## 6. LLM 抽象设计

### 6.1 核心接口

```ts
export type LlmProviderName = "mock" | "openai" | "anthropic" | "custom";

export interface LlmMessage {
  role: "system" | "user" | "assistant";
  content: string;
}

export interface LlmGenerateParams {
  model?: string;
  messages: LlmMessage[];
  temperature?: number;
  maxTokens?: number;
  responseFormat?: "text" | "json";
}

export interface LlmGenerateResult {
  provider: LlmProviderName;
  model: string;
  content: string;
  usage?: {
    inputTokens?: number;
    outputTokens?: number;
    totalTokens?: number;
  };
  raw?: unknown;
}

export interface LlmClient {
  generate(params: LlmGenerateParams): Promise<LlmGenerateResult>;
}
```

### 6.2 Provider 设计

- `mock`
  - 用于本地开发和联调
  - 可返回固定内容、模板内容或从本地 fixture 读取
- `openai`
  - 对接 OpenAI Chat / Responses 风格接口
- `anthropic`
  - 对接 Anthropic Messages API
- `custom`
  - 对接自定义兼容接口，优先兼容 OpenAI 风格

### 6.3 工厂模式

```ts
export interface LlmFactory {
  create(provider: LlmProviderName): LlmClient;
}
```

### 6.4 设计原则

- 所有工作流只依赖 `LlmClient`，不直接依赖具体厂商 SDK
- 所有 provider 输出统一为文本或 JSON 字符串
- 对需要结构化回写的步骤，先要求 LLM 返回 JSON，再用 `zod` 校验
- `mock` provider 要支持稳定复现，便于测试 CLI 流程
- 每次 LLM 调用都需要记录日志，至少包含耗时和成功状态

### 6.5 LLM 日志设计

每次 `generate` 调用建议打印三类日志：

- 开始日志
  - `event=llm.call.start`
- 成功日志
  - `event=llm.call.finish`
- 失败日志
  - `event=llm.call.error`

建议字段：

- `traceId`
- `runId`
- `command`
- `provider`
- `model`
- `messageCount`
- `promptLength`
- `responseFormat`
- `success`
- `durationMs`
- `usage.inputTokens`
- `usage.outputTokens`
- `usage.totalTokens`
- `contentLoggingEnabled`

日志原则：

- 不打印 API Key
- 默认不打印 Prompt 和响应正文全文
- 当 `LOG_LLM_CONTENT_ENABLED=true` 或命令行显式开启时，允许记录输入输出内容
- 即使开启，也要按 `LOG_LLM_CONTENT_MAX_CHARS` 截断
- 异常时要记录错误类型、错误消息、重试次数

AI 输入输出内容日志建议：

- 输入日志
  - `event=llm.call.input`
  - 记录 `messages` 的截断内容或合并摘要
- 输出日志
  - `event=llm.call.output`
  - 记录模型返回内容的截断文本

内容日志开启方式建议：

- 全局开启
  - 通过 `.env` 设置 `LOG_LLM_CONTENT_ENABLED=true`
- 单次开启
  - 在命令上增加 `--log-llm-content`

例如：

```bash
novel draft --book 1 --chapter 12 --log-llm-content
novel review --book 1 --chapter 12 --log-llm-content
```

## 7. 数据库抽象设计

### 7.1 目标

- 当前使用 SQLite
- 后续支持 MySQL 时，不改业务层工作流
- 业务层只依赖仓储接口和查询服务接口

### 7.2 分层

```text
CLI Command
  -> Workflow Service
    -> Repository / Query Service
      -> Database Client
        -> Kysely Dialect (SQLite / MySQL)
```

### 7.3 核心接口建议

```ts
export type DbClientName = "sqlite" | "mysql";

export interface DatabaseManager {
  getClient(): Kysely<Database>;
  migrate(): Promise<void>;
  destroy(): Promise<void>;
}
```

### 7.4 Repository 约束

- 每个聚合根至少有一个 repository
- repository 不关心 LLM，只负责数据读写
- 跨表召回可单独放在 query service，而不是堆进单个 repository
- 所有增删改查都要打印日志

### 7.5 数据库日志设计

范围要求：

- `create`
- `list`
- `get`
- `update`
- `delete`
- 复杂召回查询

建议事件名：

- `db.create`
- `db.list`
- `db.get`
- `db.update`
- `db.delete`
- `db.query`

建议字段：

- `traceId`
- `runId`
- `command`
- `table`
- `operation`
- `entityType`
- `entityId`
- `bookId`
- `filters`
- `changedFields`
- `rowCount`
- `success`
- `durationMs`

说明：

- `list/get/query` 也需要日志，满足“数据表增删改查都打印日志”
- `filters` 和 `changedFields` 只打印摘要，不直接全量打印超长文本
- 失败日志要带上错误类型和错误消息

### 7.6 MySQL 预留方式

- 所有表结构避免依赖 SQLite 私有特性
- JSON 字段尽量定义为 `text` 存储 JSON 字符串，再由业务层序列化/反序列化
- 自增主键统一用 `id`
- 时间字段统一 `created_at`、`updated_at`
- 迁移文件按通用 SQL/`kysely` schema builder 编写

## 8. 核心数据模型

### 8.1 通用字段约定

多数表建议包含以下字段：

- `id`
- `book_id`，用于归属同一本书
- `created_at`
- `updated_at`

需要补充信息能力的表，统一增加：

- `append_notes`
  - 文本型
  - 用于追加最新变化、状态补充、临时备注
- `keywords`
  - JSON 字符串数组，如 `["宗门", "灵石", "剑修"]`
  - 单个关键词不超过 8 个汉字
  - 用于章节规划和正文召回

### 8.2 关键词规则

- 每个词建议 1 到 8 个汉字
- 避免整句、避免空格长短语
- 每个实体建议 3 到 15 个关键词
- 入库前做长度校验与去重

## 9. 表设计

以下字段以 V1 可用为优先，后续可以增补。

### 9.1 书籍表 `books`

用途：一本书的顶层容器。

建议字段：

- `id`
- `title` 书名
- `summary` 本书简介
- `target_chapter_count` 预期章节数
- `current_chapter_count` 当前已批准章节数
- `status` 状态，如 `planning | writing | completed | archived`
- `metadata` 扩展 JSON
- `created_at`
- `updated_at`

说明：

- 章节列表不直接冗余存储在书籍表中，通过 `chapters.book_id` 关联

### 9.2 大纲表 `outlines`

用途：分卷、分章规划，以及连续章节剧情主线。

建议字段：

- `id`
- `book_id`
- `volume_no` 分卷编号
- `volume_title` 分卷标题
- `chapter_start_no` 起始章节号
- `chapter_end_no` 结束章节号
- `outline_level` 枚举：`volume | chapter_arc`
- `title` 大纲标题
- `story_core` 故事核心剧情
- `main_plot` 主线
- `sub_plot` 支线
- `foreshadowing` 伏笔
- `expected_payoff` 预期回收点
- `notes`
- `created_at`
- `updated_at`

说明：

- 一条大纲记录可以表示某个分卷，或者一段连续章节弧线
- `chapter_start_no + chapter_end_no` 用于定义连续章节范围

### 9.3 世界设定表 `world_settings`

用途：存储世界规则和基础系统设定。

- `age`

  - 支持多个职业
- `levels`
  - JSON 字符串数组

- `currencies`

  - 示例：`[{ "type": "金币", "amount": 120 }, { "type": "灵石", "amount": 30 }]`

### 9.6 关系表 `relations`

用途：统一表示人物与人物、势力与势力、人物与势力的关系。

建议字段：

- `id`
- `book_id`
- `source_type` 枚举：`character | faction`
- `source_id`
- `target_type` 枚举：`character | faction`
- `target_id`
- `relation_type`

- `description`
- `append_notes`
- `keywords`
- `created_at`
- `updated_at`

说明：

- 这个设计天然支持三种关系：
  - 人物 <-> 人物

- `chapters.current_review_id` 指向当前使用中的审阅版本
- 若后续支持人工 review 导入，也可以直接复用该表

### 9.13 正式文稿表 `chapter_finals`

用途：存储章节正式稿，每次 `approve` 或手工导入正式稿都新增一条。

建议字段：

- `id`
- `book_id`
- `chapter_id`
- `chapter_no`
- `version_no`
- `based_on_draft_id`
- `status`
  - 枚举：`active | archived`
- `content`
- `summary`
- `word_count`
- `source_type`
  - 枚举：`approved | imported | manual`
- `created_at`
- `updated_at`

说明：

- `chapters.current_final_id` 指向当前使用中的正式稿版本
- `approve` 不再直接覆盖章节主表正文，而是生成新的正式稿版本

## 10. 推荐建表顺序

1. `books`
2. `outlines`
3. `world_settings`
4. `items`
5. `characters`
6. `factions`
7. `relations`
8. `story_hooks`
9. `chapters`
10. `chapter_plans`
11. `chapter_drafts`
12. `chapter_reviews`

原因：

- `books` 是所有表的归属源

## 11. 召回设计

### 11.1 目标

在 `plan`、`draft`、`review`、`repair` 阶段，为 AI 提供和当前章节最相关的上下文，而不是全量喂入。

### 11.2 召回来源

- 当前书籍基本信息
- 当前章节附近的大纲
- 当前章节之前 1 到 3 章摘要或 final
- 当前未回收或正在推进的 `story_hooks`
- 通过关键词匹配命中的人物、势力、物品、关系、世界设定
- 上一章 `approve` 后产生的增量补充信息

### 11.3 V1 召回策略

- 显式关联优先
  - 先用章节表中 `actual_*_ids`
- 关键词匹配兜底
  - 从计划内容、草稿内容、章节标题中抽取词
- 状态过滤
  - 优先召回 `active`、`open`、`progressing` 的内容
- 数量控制
  - 每类实体限制返回前 N 条，避免上下文爆炸

### 11.4 `plan` 阶段固定召回模板

建议 `plan` 命令在喂给 AI 前，把召回上下文整理成固定分区，而不是每次临时拼接。

推荐结构：

1. 书籍信息
2. 本章相关大纲
3. 上文章节延续
4. 未完成故事钩子
5. 相关人物
6. 相关势力
7. 相关物品
8. 关键关系
9. 世界规则与系统约束
10. 最新增量补充信息
11. 本章规划风险提醒

各分区建议内容：

- 书籍信息
  - 书名
  - 本书简介
  - 预期章节数
  - 当前章节号
- 本章相关大纲
  - 所属分卷
  - 当前章节区间对应的大纲
  - `story_core`
  - `main_plot`
  - `sub_plot`
  - `foreshadowing`
  - `expected_payoff`
- 上文章节延续
  - 前 1 到 3 章标题
  - 前 1 到 3 章摘要
  - 前章末尾的重要状态变化
- 未完成故事钩子
  - `open/progressing` 状态的 hook
  - 钩子来源章节
  - 预期触发或回收章节
  - 当前推进状态
- 相关人物
  - 姓名
  - 当前身份/职业/等级
  - 当前状态
  - 当前目标
  - 与本章相关的最新 `append_notes`
- 相关势力
  - 势力名称
  - 势力类型
  - 核心目标
  - 当前状态
  - 与本章相关的最新 `append_notes`
- 相关物品
  - 名称
  - 类别
  - 当前归属
  - 稀有度
  - 与本章相关的最新 `append_notes`
- 关键关系
  - 关系双方
  - 关系类型
  - 当前强度
  - 最近变化
- 世界规则与系统约束
  - 世界规则
  - 职业体系
  - 货币体系
  - 战力体系
  - 涉及地区/国家规则
- 最新增量补充信息
  - 优先汇总上一章 `approve` 写入的 `append_notes`
  - 用于提醒模型最近发生但尚未沉淀到主字段的事实
- 本章规划风险提醒
  - 不可违背的设定
  - 不可跳过的钩子
  - 不可写错的人物/物品归属

### 11.5 `plan` 召回优先级建议

建议按以下顺序构造上下文：

1. 书籍信息
2. 当前章节命中的大纲
3. 上一章到前三章的延续信息
4. 当前未回收钩子
5. 显式关联实体
6. 关键词召回实体
7. 世界规则补充
8. 最新增量补充信息
9. 风险提醒

### 11.6 `plan` 单次召回数量建议

为避免上下文过大，V1 可先限制：

- 大纲：`1-3` 条
- 历史章节：`1-3` 章
- 故事钩子：`5-10` 条
- 人物：`5-12` 个
- 势力：`3-8` 个
- 物品：`3-8` 个
- 关系：`5-10` 条
- 世界设定：`3-8` 条
- 增量补充：每类取最近 `3-5` 条

### 11.7 `plan` 关键词驱动召回策略

`plan` 阶段建议加入“作者意图 -> 关键词 -> 强相关召回”的链路。

推荐输入来源：

- 作者主动输入的本章意图
- 用户手工指定的关联实体 ID
- 当前章节命中的大纲
- 前 1 到 3 章摘要

推荐流程：

1. 若用户传入作者意图，优先使用该意图
2. 若用户未传入作者意图，则基于近期大纲和前几章摘要，让 AI 先生成一个“本章作者意图草案”
3. 基于作者意图草案，让 AI 提取本章关键词
4. 将用户手工指定的实体 ID 直接加入强相关集合
5. 再用关键词匹配人物、势力、物品、关系、钩子、世界设定
6. 按相关性打分筛选强相关数据
7. 将最终召回结果写入新的 `chapter_plans` 记录

关键词提取建议输出：

- `intent_summary`
  - 本章作者意图摘要
- `keywords`
  - 本章关键词数组
- `must_include`
  - 必须出现的人物/势力/物品/钩子
- `must_avoid`
  - 明确不应违背的设定或情节

强相关判定建议：

- 手工指定 ID
  - 直接判定为强相关
- 命中当前章节大纲范围
  - 高优先级
- 关键词命中数量 >= 2
  - 高优先级
- `open/progressing/active` 状态且命中关键词
  - 高优先级
- 最近 1 到 3 章实际出现过且命中关键词
  - 高优先级

- `stage=plan`

- `stage=final`
  - 导出 `chapters.current_final_id` 指向的 `chapter_finals.content`

```md
## Content

```

- 解析 YAML Front Matter
- 校验 `book_id`、`chapter_no`、`stage` 是否与 CLI 参数一致
- `title`、`summary`、`word_count` 可作为可更新字段
- `## Content` 后的正文作为目标阶段正文内容写回数据库

字段写回建议：

- `stage=plan`
  - 新增一条 `chapter_plans`
  - 并更新 `chapters.current_plan_id`
- `stage=draft`
  - 新增一条 `chapter_drafts`
  - 并更新 `chapters.current_draft_id`
- `stage=final`
  - 新增一条 `chapter_finals`
  - 并更新 `chapters.current_final_id`
- 若 Markdown 中有新的 `title` 或 `summary`，同步更新章节表对应字段
- `word_count` 可优先根据正文自动重算，再覆盖旧值

状态联动建议：

- 导入 `plan`
  - 若原状态为 `todo`，可更新为 `planned`
- 导入 `draft`
  - 若原状态为 `planned` 或 `reviewed` 或 `repaired`，统一更新为 `drafted`
- 导入 `final`
  - 默认仅允许在 `approved` 后修订正式稿
  - 若要提前导入 `final`，需要显式 `--force`

V1 建议限制：

- `chapter_reviews` 先不走 Markdown 导入导出
- `approve` 后若用户手改 `final` 并重新导入，只更新章节正式文稿，不自动二次回写人物/势力/钩子
- 如果需要二次回写，后续可增加 `novel approve --resync`
- `plan/draft` 的手工导入都应保留版本历史，而不是覆盖旧记录
- `final` 的手工导入也应保留版本历史，而不是覆盖旧记录

### 12.3 `plan`

目标：

- 为指定章节生成章节规划
- 召回相关设定、大纲、人物、势力、物品、钩子
- 输出给后续 `draft` 使用的结构化章节计划

输入：

- `book_id`
- `chapter_no`
- 可选 `--authorIntent`
- 可选 `--characterIds`
- 可选 `--factionIds`
- 可选 `--itemIds`
- 可选 `--hookIds`
- 可选 `--worldIds`
- 可选 `--relationIds`
- 可选 `--model`
- 可选 `--force`

输入说明：

- `--authorIntent`
  - 作者直接描述本章想写什么
- `--characterIds`
  - 手工指定本章重点人物 ID，多个可逗号分隔
- `--factionIds`
  - 手工指定本章重点势力 ID
- `--itemIds`
  - 手工指定本章重点物品 ID
- `--hookIds`
  - 手工指定本章必须关注的钩子 ID
- `--worldIds`
  - 手工指定本章必须参考的世界设定 ID
- `--relationIds`
  - 手工指定本章必须参考的关系 ID

处理流程：

1. 读取书籍信息
2. 读取命中的大纲范围
3. 读取上文章节摘要和近期已批准章节信息
4. 读取用户手工指定的关联实体
5. 若用户传入 `authorIntent`，直接作为本章作者意图输入
6. 若用户未传入 `authorIntent`，则依据近期大纲和前几章摘要，让 AI 先生成本章作者意图草案
7. 基于作者意图，让 AI 提取本章关键词、必须包含项、风险项
8. 将手工指定实体直接纳入强相关候选集合
9. 用关键词去匹配人物、势力、物品、关系、钩子、世界设定
10. 结合大纲命中、近期章节、状态、显式指定 ID 做强相关筛选
11. 按固定召回模板整理上下文分区
12. 组织 Prompt 请求 LLM 生成章节规划
13. 记录作者意图、关键词、召回摘要和 LLM 调用日志
14. 新增一条 `chapter_plans`
15. 更新 `chapters.current_plan_id`
16. 如有旧计划版本，可将其标记为 `archived`
17. 更新章节状态为 `planned`

建议输出结构：

- 作者意图
- 提取关键词
- 手工指定关联实体
- 强相关召回结果
- 本章目标
- 本章冲突
- 本章推进的主线
- 本章推进的支线
- 本章应触发/回收的钩子
- 出场人物
- 出场势力
- 关键道具
- 章节节奏分段
- 风险提醒

建议 `chapter_plans.content` 结构额外包含：

- `author_intent`
  - 用户输入或 AI 生成的作者意图
- `intent_source`
  - `user_input | ai_generated`
- `intent_keywords`
  - AI 提取出的关键词数组
- `manual_entity_refs`
  - 用户手工指定的实体引用
- `retrieved_context`
  - 本次强相关召回结果

`retrieved_context` 建议结构：

```ts
interface PlanRetrievedContext {
  characters: Array<{ id: number; name: string; reason: string; content: string }>;
  factions: Array<{ id: number; name: string; reason: string; content: string }>;
  items: Array<{ id: number; name: string; reason: string; content: string }>;
  hooks: Array<{ id: number; title: string; reason: string; content: string }>;
  relations: Array<{ id: number; reason: string; content: string }>;
  worldSettings: Array<{ id: number; title: string; reason: string; content: string }>;
}
```

说明：

- 每条召回结果都应记录 `id`
- 每条召回结果都应记录 `reason`
  - 例如 `manual_id`、`keyword_hit`、`recent_chapter`、`outline_hit`
- `content` 用于保留喂给 AI 的核心信息摘要，便于后续 `draft/review` 复用
- 若 `author_intent` 为空，`chapter_plans.content` 中应明确保留 AI 自动生成的意图文本，避免后续不可追踪

### 12.4 `draft`

目标：

- 基于当前 plan 版本生成章节草稿

输入：

- `book_id`
- `chapter_no`
- 可选 `--model`
- 可选 `--targetWords`

处理流程：

1. 检查本章是否已有 `chapters.current_plan_id`
2. 读取当前 `chapter_plans` 记录
3. 召回相关实体和上文上下文
4. 组织 Prompt 请求 LLM 写草稿
5. 新增一条 `chapter_drafts`
6. 更新 `chapters.current_draft_id`
7. 如有旧草稿版本，可将其标记为 `archived`
8. 更新章节状态为 `drafted`

### 12.5 `review`

目标：

- 检查本章草稿的合理性、一致性和潜在 bug

检查项建议：

- 是否违背世界设定
- 人物行为是否前后不一致
- 势力关系是否冲突
- 物品归属是否错误
- 钩子是否遗忘或误回收
- 与大纲是否偏离
- 章节节奏和信息量是否失衡

输入：

- `book_id`
- `chapter_no`
- 可选 `--model`

处理流程：

1. 检查是否有 `chapters.current_draft_id`
2. 读取当前 `chapter_drafts`
3. 读取当前 `chapter_plans`
4. 召回相关设定、上文、hook、实体
5. 组织 Prompt 请求 LLM 做审阅
6. 输出结构化问题清单
7. 新增一条 `chapter_reviews`
8. 更新 `chapters.current_review_id`
9. 如有旧 review 版本，可将其标记为 `archived`
10. 更新章节状态为 `reviewed`

建议 `chapter_reviews.raw_result` 结构：

- `summary`
- `issues`
- `risks`
- `continuity_checks`
- `repair_suggestions`

### 12.6 `repair`

目标：

- 基于当前草稿和当前 review 生成新草稿版本

输入：

- `book_id`
- `chapter_no`
- 可选 `--model`

处理流程：

1. 检查是否已存在 `chapters.current_draft_id` 和 `chapters.current_review_id`
2. 读取当前 `chapter_drafts`
3. 读取当前 `chapter_reviews`
4. 让 LLM 依据问题清单修复文本
5. 新增一条 `chapter_drafts`
6. 将新草稿设为 `chapters.current_draft_id`
7. 可将旧草稿标记为 `archived`
8. 更新章节状态为 `repaired`

V1 建议：

- 不直接覆盖旧草稿
- 通过 `chapter_drafts` 原生保留版本历史

### 12.7 `approve`

目标：

- 将当前草稿批准为正式章节
- 并根据最终正文同步更新设定库

输入：

- `book_id`
- `chapter_no`
- 可选 `--model`
- 可选 `--dry-run`

处理流程：

1. 检查本章是否已有可批准文本
2. 读取 `chapters.current_draft_id` 指向的草稿版本
3. 新增一条 `chapter_finals`
4. 更新 `chapters.current_final_id`
5. 更新章节状态为 `approved`
6. 基于当前 `chapter_finals` + 当前 plan + 当前 review 提取结构化事实变更
7. 更新相关实体表
8. 对无法覆盖原字段但需要保留的新事实写入 `append_notes`
9. 更新 `actual_*_ids`
10. 更新 `books.current_chapter_count`

`approve` 的核心职责不是只改章节状态，而是做“事实落库”。

## 13. `approve` 回写策略

### 13.1 可以直接更新的字段

例如：

- 人物当前位置
- 人物状态
- 势力状态
- 物品当前归属
- 钩子状态

### 13.2 应 append 的信息

例如：

- 某人物新增一段经历，但不适合覆盖原背景
- 某势力最新动向
- 某物品新用途、新限制
- 某故事钩子的新线索

写入策略建议：

- 在 `append_notes` 追加一段带章节来源的文本
- 示例：`[Chapter 12] 林夜确认拥有黑铁令，可临时调动外门弟子。`

### 13.3 可能新增的实体

`approve` 时要支持从最终正文中发现并新增：

- 新人物
- 新势力
- 新物品
- 新故事钩子

建议流程：

1. 让 LLM 输出结构化 diff
2. 使用 `zod` 校验
3. 对已存在实体做匹配
4. 匹配不到时新增记录

```ts
interface ApproveDiff {
```

## 15. 工作流服务设计

建议每个命令对应一个 workflow service：

- `PlanChapterWorkflow`
- `DraftChapterWorkflow`
- `ReviewChapterWorkflow`
- `RepairChapterWorkflow`
- `ApproveChapterWorkflow`

资源 CRUD 建议对应独立 application service：

- `BookService`
- `OutlineService`
- `WorldSettingService`
- `CharacterService`
- `FactionService`
- `RelationService`
- `ItemService`
- `StoryHookService`
- `ChapterService`

每个 workflow 保持同样结构：

1. 参数校验
2. 加载上下文
3. 调用 LLM
4. 校验输出
5. 写库
6. 返回 CLI 结果

日志要求：

- 每个 workflow step 都建议打印开始和结束日志
- `plan/draft/review/repair/approve` 至少记录总耗时
- 召回步骤建议记录命中数量和各类实体条数
- LLM 步骤建议记录模型、耗时、成功与否
- 当 `logLlmContent=true` 时，额外打印 LLM 输入输出内容日志
- 写库步骤建议记录更新表、主键、变更字段摘要
- `approve` 额外记录新增实体数量、更新实体数量、append 次数

## 17. 错误处理建议

### 17.1 LLM 层

- Provider 未配置 API Key
- 接口超时
- 返回格式不合法
- JSON 解析失败

### 17.2 数据层

- 指定 `book_id` 不存在
- 指定章节不存在
- 批准阶段实体 ID 找不到
- 迁移未执行

### 17.3 工作流层

- `draft` 前没有 `plan`
- `review` 前没有 `draft`
- `repair` 前没有 `review`
- `approve` 前没有可批准文本

3. 之后接入 `openai` 和 `anthropic`

## 21. 结论

这个 V1 方案的核心思想是：

- 用结构化设定库管理小说事实
- 用统一工作流驱动章节生产
- 用 `approve` 把正文事实回灌到设定库
- 用 LLM 和数据库都做抽象，为后续 provider 与 MySQL 扩展留口子

如果按这个方案实现，后续继续扩展 Web 界面、向量召回、版本系统和多项目管理都会比较顺滑。
