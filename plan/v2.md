# AI 小说工具 V2 方案

## 1. 背景

V1 已经完成以下核心能力：

- 本地 CLI 工作流：`plan -> draft -> review -> repair -> approve`
- 多 provider LLM 抽象：`mock`、`openai`、`anthropic`、`custom`
- 基于规则式匹配的设定召回与 `retrieved_context` 固化
- 章节版本化存储：`chapter_plans`、`chapter_drafts`、`chapter_reviews`、`chapter_finals`
- `approve` 后结构化事实回写
- Markdown 导入导出
- 基础文档、日志、测试与环境变量配置体系

但从 V1 的实现与当前代码状态看，后续最值得继续投入的方向主要集中在两块：

1. **LLM 召回与上下文组织能力升级**
2. **数据库层从 SQLite 扩展到 MySQL**

因此 V2 目标不是推倒重来，而是在保留 V1 主体架构的前提下，把“可扩展性、召回质量、部署能力”向前推进一层。

---

## 2. V2 总目标

V2 建议聚焦以下目标：

- 在不破坏现有 CLI 使用方式的前提下，升级召回上下文结构与质量
- 让不同工作流阶段消费不同粒度的上下文，而不是完全共享同一份原始召回结果
- 为后续 rerank / embedding 留出清晰插口
- 补齐 MySQL 支持，做到 SQLite / MySQL 可切换
- 补齐 MySQL 下的迁移、连接、配置与测试链路
- 在工程层补充更多注释、文档与验证手段，降低后续维护门槛

### 优先级原则

V2 主线优先级建议明确为：

1. **先做召回升级**
2. **再做 MySQL 支持**
3. **最后再处理 rerank / embedding 预留**

原因：

- 召回升级直接影响生成质量和工作流稳定性
- MySQL 支持主要提升部署与扩展能力
- rerank / embedding 更适合作为 V2 后段或 V2.5 的预留项，而不是主线先决条件

---

## 3. V2 范围

### 3.1 包含

- 召回结构分层
- 阶段化上下文裁剪
- 更细的召回打分模型
- 高风险连续性实体优先召回
- 为 rerank / embedding 预留接口
- MySQL dialect 与连接能力
- MySQL 配置项与文档
- MySQL 下 migration / 初始化 / 校验能力
- 关键核心代码持续补充中文注释

### 3.2 暂不包含

- Web UI
- 多用户协作
- 向量数据库正式接入生产闭环
- 自动章节批处理调度系统
- 权限系统
- MySQL 专属特性优化（例如方言特化 JSON 能力、专用索引策略）

---

## 4. V2 设计主题

## 4.1 主题一：召回结果从“单层结果”升级为“分层上下文”

V1 中 `retrieved_context` 已经可以稳定工作，但当前主要问题是：

- `plan` 生成出来的上下文会被后续多个阶段直接复用
- 不同阶段实际需要的上下文密度不同
- 召回结果里“硬约束”和“软参考”尚未显式分层

V2 建议把召回结果升级为更明确的结构，例如：

- `hardConstraints`
  - 人物状态
  - 关键关系
  - 世界规则
  - 物品归属
  - 钩子状态
- `softReferences`
  - 近期章节摘要
  - 较弱相关设定
  - 辅助背景信息
- `riskReminders`
  - 连续性高风险项
  - 未回收钩子
  - 容易写错的设定边界

目标：

- `draft` 更清楚什么不能写错
- `review` 更清楚什么必须核对
- `approve diff` 更清楚什么属于事实边界，什么只是参考

---

## 4.2 主题二：从“统一上下文复用”升级为“阶段化视图”

V1 的一个优点是稳定：

- `draft / review / repair / approve` 都围绕同一份 `chapter_plans.retrieved_context`

但缺点也很明显：

- `draft` 需要写作上下文
- `review` 更需要核对上下文
- `approve diff` 更需要结构化事实参照

V2 建议：

- 仍然保留 `chapter_plans.retrieved_context` 作为基础来源
- 但在进入不同工作流前，增加一层上下文裁剪器，例如：
  - `buildDraftContextView()`
  - `buildReviewContextView()`
  - `buildApproveDiffContextView()`

目标：

- 不重新检索，避免漂移
- 但按阶段压缩和整理上下文
- 减少 prompt 膨胀与噪声

---

## 4.3 主题三：召回打分从“统一公式”升级为“实体差异化权重”

V1 当前主要采用统一公式：

- 手工 ID：高优先级
- 关键词命中：基础加分
- 少量业务加权（如钩子邻近性、关系手工连接）

V2 建议把权重更细化：

### 人物
- `name`、`alias` 权重更高
- `goal`、`background` 次高
- 长文本附注命中权重相对更低

### 势力
- `name`、`category`、`core_goal` 权重不同

### 物品
- `name`、`description`、`rarity` 权重不同

### 关系
- 关系两端实体与 `relationType` 应有更高优先级

### 世界设定
- `title`、`category` 权重高于长文本正文

目标：

- 减少“命中了长文本但其实不够关键”的噪声召回
- 提高人物、规则、关键关系这类强约束内容的排序质量

---

## 4.4 主题四：增加“高风险连续性实体”优先召回

有些内容不一定最相关，但一旦写错，代价最高。

V2 建议引入一类高风险连续性实体：

- 当前地点中的人物
- 当前持有中的关键物品
- 关键关系状态
- 未回收重要钩子
- 特殊世界规则

这些内容可以在排序时有额外提升，或单独进入 `riskReminders` / `hardConstraints`。

目标：

- 减少后续章节常见连续性错误
- 提升 `review` 与 `approve` 阶段的约束强度

---

## 4.5 主题五：为 rerank / embedding 留出口

V1 目前属于规则式召回，优点是：

- 稳定
- 可解释
- 易调试

V2 不建议直接让 embedding 接管主召回，而建议分阶段演进：

### 定位

这部分建议视为：

- **V2 optional**，或
- **V2.5 预研项**

不应与召回结构升级、MySQL 支持争夺主线优先级。

### V2.1
- 规则式召回仍然作为主流程
- 增加 rerank 接口
- 候选集先由规则召回得到，再做二次重排

### V2.2（预留）
- embedding 作为补充召回源
- 最终仍然交给业务层 rerank + 截断

建议新增抽象：

- `RetrievalCandidateProvider`
- `RetrievalReranker`
- `ContextViewBuilder`

目标：

- 未来扩展 embedding 不必重写工作流
- 保持现有 explainability

---

## 4.6 主题六：支持 MySQL

当前代码状态：

- `src/core/db/dialects/mysql.ts` 仍是未实现占位
- `src/core/db/client.ts` 仅支持 `sqlite`
- `.env.example` 已声明 `DB_CLIENT: sqlite | mysql`
- 数据表设计上已考虑 SQLite / MySQL 兼容，JSON 字段统一按 `TEXT` 存储

这说明 V1 已经有“面向 MySQL 预留”的意识，但尚未真正落地。

V2 建议正式支持 MySQL，目标包括：

- 可以通过 `DB_CLIENT=mysql` 启动
- 提供必要的 MySQL 连接配置
- 迁移逻辑在 MySQL 下可运行
- CRUD、workflow、JSON 字段策略在 MySQL 下保持行为一致

### MySQL 支持边界

V2 对 MySQL 的目标应明确为：

- **让现有 SQLite 业务链路在 MySQL 下等价可运行**
- **不是在 V2 内引入 MySQL 专属能力或方言优化**

这样可以控制范围，避免 MySQL 支持过程中演变成数据库层重设计。

---

## 5. MySQL 支持详细方案

## 5.1 配置扩展

建议在 `.env.example` 和 `src/config/env.ts` 中补充：

```dotenv
DB_CLIENT=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_NAME=myai_novel
DB_USER=root
DB_PASSWORD=
DB_POOL_MIN=0
DB_POOL_MAX=10
```

说明：

- SQLite 保持单文件本地开发默认值
- MySQL 提供最小连接配置
- 后续 `createDatabaseManager()` 根据 `DB_CLIENT` 选择 dialect

---

## 5.2 Dialect 实现

建议实现：

- `src/core/db/dialects/mysql.ts`

做法：

- 使用 Kysely 的 MySQL dialect
- 接入兼容驱动（按项目实际选择合适库）
- 提供连接池配置
- 保持与 SQLite 侧 `createSqliteDb()` 同样的返回接口

---

## 5.3 DatabaseManager 扩展

当前：

- `src/core/db/client.ts` 里如果 `env.DB_CLIENT !== "sqlite"` 会直接报错

V2：

- 改成根据 `env.DB_CLIENT` 路由：
  - `sqlite` -> `createSqliteDb()`
  - `mysql` -> `createMysqlDb()`
- 日志里补充 MySQL 连接目标
- `destroy()` 正确释放 MySQL 连接池

---

## 5.4 Migration 兼容性

V1 migration 已按“JSON 存 TEXT”策略设计，这对 MySQL 是有利的。

V2 需要补充：

- 检查所有建表语句是否依赖 SQLite 特有行为
- 检查索引、默认值、时间字段、唯一约束是否在 MySQL 下兼容
- 为 migration 增加 MySQL 下的验证用例

重点检查：

- DATETIME 默认值策略
- 自增主键语法
- `TEXT` 字段索引限制
- 布尔语义与字符串布尔解析

---

## 5.5 Repository / Workflow 兼容性核查

V2 不应为了支持 MySQL 重写业务层，而应优先验证当前 Repository 和 Workflow 是否已经方言无关。

重点核查：

- 查询构造是否依赖 SQLite 特定语法
- transaction 使用方式在 MySQL 下是否一致
- JSON 文本字段的序列化 / 反序列化是否仍然稳定
- chapter pointers、版本号查询、去重逻辑在 MySQL 下是否保持一致

---

## 5.6 MySQL 测试建议

建议新增至少两层测试：

### 单元/集成测试
- MySQL 环境变量解析
- MySQL client 创建
- migration 可执行

### 端到端测试
- `db init`
- `book create`
- `chapter create`
- `plan`
- `draft`
- `review`
- `repair`
- `approve`
- `chapter export/import`

目标：

- 确认 V1 全链路在 MySQL 下可跑通

## 8. V2 分阶段建议

## 阶段一：召回结构升级（P0）

目标：先提升生成质量与上下文稳定性。

包含：

- `retrievedContext` 分层
- 阶段化上下文视图
- Prompt / Workflow 适配
- SQLite 回归验证

## 阶段二：召回质量增强（P1）

目标：在结构稳定后，再提升召回质量。

包含：

- 打分逻辑拆分
- 差异化权重
- 高风险连续性实体优先
- 召回质量验证

## 阶段三：MySQL 支持落地（P2）

目标：让当前业务链路在 MySQL 下等价跑通。

包含：

- MySQL 技术选型
- `env` 扩展
- `mysql` dialect 实现
- `client` 路由扩展
- migration / repository / workflow 兼容性验证
- MySQL 端到端测试

## 阶段四：扩展预留与收尾（P3）

目标：为未来能力预留接口，并完成文档收口。

包含：

- rerank / embedding 预留接口
- 文档全面补齐
- 注释与维护性收尾

---

## 9. V2 成功标准

如果 V2 完成，建议至少满足以下标准：

### 召回侧
- `retrievedContext` 已分层
- `draft / review / approve diff` 使用不同上下文视图
- 高风险连续性项可显式进入约束层
- 召回结果更易解释和调试

### 数据库侧
- `DB_CLIENT=mysql` 可用
- `db init` / `db check` / 迁移在 MySQL 下可运行
- 核心 workflow 在 MySQL 下可通过
- SQLite 与 MySQL 业务行为保持一致
- 默认 SQLite 开发体验不变

### 工程侧
- 关键核心代码已有中文注释
- 文档可支撑新接手开发者理解主要链路
- 关键环境变量和数据库差异已文档化
- 默认规则式召回行为仍可用，不依赖 rerank / embedding

---

## 10. V2 优先级建议

如果只能先做最重要的几项，我建议顺序如下：

1. **召回分层与阶段化上下文视图**
2. **Prompt / workflow 接新结构**
3. **高风险连续性实体优先召回**
4. **MySQL dialect + client 路由 + env 扩展**
5. **MySQL 端到端验证**
6. **召回差异化权重**
7. **rerank 预留接口**

---

## 11. 一句话总结

V2 的核心不是增加更多命令，而是把 V1 已经跑通的章节工作流，升级成一套 **召回更准、上下文更稳、数据库更可扩展** 的基础设施。
