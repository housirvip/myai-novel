# AI 小说工具 V3 方案

## 1. 背景

V2 已经完成以下关键基础：

- `retrieved_context` 分层为 `hardConstraints` / `softReferences` / `riskReminders`
- `draft / review / repair / approve` 已使用阶段化上下文视图
- 规则式召回、解释信息、MySQL 支持与基础测试链路已经收口
- 已为 `RetrievalCandidateProvider` / `RetrievalReranker` 预留扩展接口

当前默认主链路的真实状态是：

- **已有规则式 ranking**：规则召回后会做打分排序与截断
- **默认仍不启用真实 rerank**：当前默认仍是 `DirectPassThroughReranker`，但已支持通过配置切换到 `HeuristicReranker`
- **embedding 已有实验链路**：已具备摘要构造、文档构造、in-memory / hybrid searcher、candidate merge、benchmark 实验；默认仍未接入正式向量后端

这意味着当前系统已经具备继续优化“拿什么信息给模型”和“怎么把信息给模型”的条件。

V3 不再把重点放在数据库能力或新工作流上，而是集中解决一个更直接影响生成质量的问题：

**让 AI 在每个阶段都拿到最准确、最重要、最可执行的信息。**

## 2. V3 总目标

V3 聚焦两条主线：

1. **召回数据优化**
2. **提示词优化**

具体目标：

- 从“能召回”升级到“优先召回最重要、最容易写错的信息”
- 从“原始信息直接塞进 prompt”升级到“按阶段、按职责、按风险组织 prompt 输入”
- 明确哪些信息必须进入模型上下文，哪些信息只能作为补充参考
- 为后续 embedding / rerank 正式接入提供稳定评测基线
- 建立一套可复现的 retrieval + prompt benchmark，避免后续优化只能凭感觉

---

## 3. 核心原则

### 3.1 正确性优先于覆盖率

比起“塞更多信息”，V3 更强调：

- 不遗漏高风险硬约束
- 不把低价值背景噪声挤进上下文
- 不让次要信息覆盖关键事实

### 3.2 重要性优先于相似性

一个信息即使和当前章节语义相关，如果它不是：

- 当前章节必需承接的事实
- 容易写错的连续性信息
- 会直接影响剧情推进的实体

也不应进入高优先级上下文。

### 3.3 Prompt 必须消费“整理后的事实”，不是直接消费“数据库原文”

数据库里适合存储的信息，不一定适合直接给模型读。

V3 需要在召回和 prompt 之间增加一层“上下文整理”，把：

- 原始字段
- 结构化状态
- 最近变更
- 风险信号

转换成更适合模型理解和执行的上下文块。

### 3.4 每个阶段只看自己真正需要的信息

- `plan` 关心本章要推进什么，以及不能碰坏什么
- `draft` 关心写作时必须遵守的约束与当前承接状态
- `review` 关心核对基准与高风险差错点
- `approve diff` 关心事实变化边界与可回写性

## 5. V3 设计主题

## 5.1 主题一：召回结果从“实体列表”升级为“事实包”

V2 的主要输出仍然是按实体类型组织的列表。这样已经够稳定，但仍有几个问题：

- 单个实体的 `content` 往往是字段拼接，信息密度不均匀
- 模型需要自己判断哪些字段重要
- 同一实体的“身份信息”“当前状态”“风险点”“与本章关系”混在一起

V3 建议把高优先级召回项升级成更明确的事实包，例如：

- `identity`
  - 实体是谁
  - 别名/称谓
  - 类型
- `currentState`
  - 当前地点
  - 当前立场
  - 当前持有
  - 当前状态
- `chapterRelevance`
  - 为什么和本章相关
  - 命中了哪些关键词/意图
  - 是否由最近章节承接而来
- `continuityRisk`
  - 容易写错什么
  - 哪个状态不能随意改
- `sourceSummary`
  - 来源于最近章节 / 大纲 / 历史事实 / 手工指定

目标：

- 让 prompt 消费的是“已整理事实”而不是原始字段串
- 让模型更容易区分“实体介绍”和“当前约束”

---

## 5.2 主题二：新增“重要性分层”而不只是“相关性排序”

V2 的核心仍然是排序后截断。V3 建议进一步把召回结果分成几层：

- `blockingConstraints`
  - 一旦写错会直接造成连续性错误
  - 必须进入 prompt
- `decisionContext`
  - 对本章剧情推进、人物决策、冲突设计有直接影响
  - 默认进入 prompt
- `supportingContext`
  - 有帮助但不是必需
  - 仅在预算允许时进入
- `backgroundNoise`
  - 命中但价值低
  - 不进入 prompt，只保留在调试信息中

这层分组不等价于 V2 的 `hardConstraints` / `softReferences`：

- `hardConstraints` 更偏“约束类型”
- V3 新分层更偏“进入 prompt 的优先级”

建议两者并存：

- `hardConstraints` 继续表达业务语义
- `blockingConstraints / decisionContext / supportingContext` 表达 prompt 预算优先级

---

## 5.3 主题三：召回规则从“字段命中”升级为“意图槽位命中”

当前关键词命中主要是文本包含。V3 建议把作者意图先拆成几个槽位，再按槽位召回：

- `subjectEntities`
  - 这章主要写谁
- `goalAndConflict`
  - 这章要推进什么目标、冲突、危机
- `locationAndScene`
  - 场景、地点、势力范围
- `plotThreads`
  - 钩子、悬念、未回收伏笔
- `forbiddenMoves`
  - 不能发生什么、不能破坏什么

然后对不同槽位匹配不同实体和字段：

### 人物
- `subjectEntities`：`name` / `alias` / 主身份标签
- `goalAndConflict`：`goal` / `personality` / 当前立场
- `locationAndScene`：`current_location`

### 势力
- `locationAndScene`：势力据点、控制范围、活动区域
- `goalAndConflict`：`core_goal` / 当前状态 / 对立关系

### 物品
- `subjectEntities`：专属名物、关键道具名
- `goalAndConflict`：用途、稀有性、归属

### 关系
- `subjectEntities`：关系两端实体
- `goalAndConflict`：`relation_type` / 当前关系张力

### 钩子
- `plotThreads`：未回收、将触发、延迟兑现

### 世界设定
- `forbiddenMoves`：规则边界、能力边界、禁忌、成本条件

目标：

- 让召回不只是“词命中了”，而是“命中了本章写作意图里的哪个职责位”
- 为后续 rerank 提供更细粒度特征

---

## 5.4 主题四：新增“最近状态”和“最后变更”优先规则

很多连续性错误不是因为没有召回到实体，而是没有把“这个实体最近刚发生了什么变化”放到最前面。

V3 建议增加两类信号：

- `recentStateSignal`
  - 最近 1 到 3 章里出现过
  - 当前状态刚发生变化
- `lastChangedFacts`
  - 位置变化
  - 关系变化
  - 物品归属变化
  - 钩子状态变化

这些信号应显式影响：

- 是否进入 `blockingConstraints`
- 是否进入 `riskReminders`
- 在 prompt 中是否以“最近变更”区块单独展示

目标：

- 减少“历史设定记住了，但最新状态没承接”的错误

---

## 5.5 主题五：新增“负面召回”与“禁止漂移”规则

V2 已有 `mustAvoid`，但它主要来自意图抽取。V3 建议补强两类负面约束：

- `doNotInvent`
  - 不要凭空新增的重要设定
- `doNotOverride`
  - 不要覆盖已知状态
- `doNotEscalate`
  - 不要无依据升级战力、财富、地位、关系

这些内容的来源可以包括：

- 世界规则
- 已有角色能力边界
- 已有关系状态
- 当前未兑现钩子
- 用户显式限制

在 prompt 中，这些约束不应埋在长 JSON 里，而应单独成为：

- `禁止改写`
- `禁止新增`
- `禁止跳步推进`

目标：

- 降低模型“合理脑补”造成的设定漂移

---

## 5.6 主题六：Prompt 从“单块 JSON”升级为“角色化上下文块”

V2 的 prompt 已经结构化，但核心上下文仍经常以 JSON 整体塞给模型。V3 建议进一步改成更适合阅读的上下文块：

- `本章必须遵守的事实`
- `本章最相关的人物与状态`
- `本章最相关的关系与冲突`
- `本章最近承接的变化`
- `本章必须推进的钩子`
- `本章禁止改写的规则`

原则：

- 关键约束用短句列表
- 解释性信息用简短摘要
- 只有调试或核对型阶段才展示更全的结构化对象

目标：

- 降低模型从 JSON 中自行提炼重点的负担
- 把 token 预算优先留给真正关键的信息

---

## 5.7 主题七：Prompt 指令从“泛约束”升级为“阶段显式职责”

V3 建议把每个阶段的提示词职责写得更硬一些。

### plan
- 先确定本章推进目标
- 再检查哪些事实不能碰
- 明确列出本章必须承接的状态

### draft
- 先遵守 `blockingConstraints`
- 再消费 `decisionContext`
- 对 `supportingContext` 保持克制

### review
- 不泛泛评价文风
- 优先检查：事实错、状态错、关系错、规则错、推进跳步

### repair
- 只修关键问题
- 不为“润色”引入新设定

### approve diff
- 只抽取真正发生变化且可落库的事实
- 不把语气词、感受描写误判成结构化变更

目标：

- 让不同阶段的模型行为更稳定，减少角色漂移

---

## 5.8 主题八：建立可量化评测，而不是只看主观观感

V3 必须建立 benchmark，否则很难判断一次召回或 prompt 改动到底是变好还是变差。

建议补三类指标：

### 召回指标
- `Constraint Recall@N`
- `Risk Coverage`
- `Important Entity Recall@N`
- `Noise Ratio`

### Prompt 指标
- `Prompt Token Budget`
- `Critical Facts Density`
- `Instruction Collision Count`

### 端到端质量指标
- `Continuity Error Rate`
- `Review Catch Rate`
- `Approve Diff Precision`

建议准备一组固定样本：

- 每个阶段 20 到 50 个真实章节案例
- 标注本章必须召回的关键事实
- 标注常见错误类型

## 6. 新的召回规则建议

## 6.1 总体流程

V3 建议将召回链路拆成六步：

1. **意图解析**
2. **候选召回**
3. **事实整理**
4. **重要性分层**
5. **阶段化裁剪**
6. **Prompt 组装**

### Step 1. 意图解析

从 `authorIntent` 中提取：

- `intentSummary`
- `subjectEntities`
- `goalAndConflict`
- `locationAndScene`
- `plotThreads`
- `mustInclude`
- `mustAvoid`

### Step 2. 候选召回

候选来源建议包含：

- 规则式字段匹配
- 手工指定实体
- 最近章节涉及实体
- 最近已变更实体
- 未回收钩子
- 当前大纲覆盖实体
- 未来 embedding 候选

### Step 3. 事实整理

对每个高优先级候选，不直接输出原始字段拼接，而是整理成：

- `entitySummary`
- `currentState`
- `recentChanges`
- `continuityRisk`
- `relevanceReasons`

### Step 4. 重要性分层

按以下规则分层：

- 若属于世界规则、已确认关系状态、当前持有、当前地点、未回收主钩子，则优先进入 `blockingConstraints`
- 若直接影响本章目标推进，则进入 `decisionContext`
- 若仅提供氛围/背景补充，则进入 `supportingContext`
- 若只是弱命中长文本，则进入 `backgroundNoise`

### Step 5. 阶段化裁剪

- `plan`：`blockingConstraints` + `decisionContext` + 少量 `supportingContext`
- `draft`：`blockingConstraints` + 与当前 scene 强相关的 `decisionContext`
- `review`：`blockingConstraints` + `recentChanges` + `riskReminders`
- `approve diff`：`blockingConstraints` + 可回写事实边界

### Step 6. Prompt 组装

优先输出短句化事实块，而不是整包 JSON。

---

## 6.2 打分模型建议

每个候选建议拆成几类分数，而不是单一总分：

- `matchScore`
  - 关键词/槽位命中程度
- `importanceScore`
  - 实体对剧情和约束的重要性
- `continuityRiskScore`
  - 写错代价有多高
- `recencyScore`
  - 最近章节里是否出现或更新
- `manualPriorityScore`
  - 用户是否显式指定

最终排序建议：

- `finalScore = matchScore + importanceScore + continuityRiskScore + recencyScore + manualPriorityScore`

同时增加保底规则：

- 世界规则和禁忌类信息，命中后尽量不被截断
- 最近一章刚变化的关键状态，命中后优先保留
- 手工指定实体默认进入高优先级层

---

## 6.3 各类实体的 V3 召回重点

### 人物
- 优先字段：`name`、`alias`、`current_location`、`goal`、`current stance`
- 强提升条件：主视角人物、最近出场、状态刚变化、与当前钩子强绑定
- 风险提醒：人设崩坏、位置错、动机错、能力越界

### 势力
- 优先字段：`name`、`category`、`core_goal`、当前控制区域、当前敌友态势
- 强提升条件：当前场景所属势力、与主冲突直接相关
- 风险提醒：势力边界错、立场错、影响力错

### 物品
- 优先字段：`name`、`owner`、`rarity`、`ability`、当前状态
- 强提升条件：当前持有、剧情目标依赖、刚转移归属
- 风险提醒：持有者错、功能错、数量错、消耗状态错

### 关系
- 优先字段：两端实体、`relation_type`、当前张力、最近变更
- 强提升条件：当前冲突核心关系、最近从敌对变合作或相反
- 风险提醒：关系跳变、情绪进度过快、历史矛盾丢失

### 钩子
- 优先字段：`title`、`status`、`trigger window`、`expected_payoff`
- 强提升条件：临近兑现、已多章铺垫、与主线直接绑定
- 风险提醒：忘记推进、过早兑现、重复兑现

### 世界设定
- 优先字段：`title`、`category`、规则边界、代价条件、禁忌
- 强提升条件：与当前地点、能力、战斗、制度直接相关
- 风险提醒：规则失效、成本消失、系统冲突

## 7. Prompt 优化建议

## 7.1 输入结构建议

V3 建议把传给模型的上下文重组为以下区块：

1. `本章目标`
2. `必须遵守的事实`
3. `最近承接的变化`
4. `本章核心人物/势力/关系`
5. `必须推进的钩子`
6. `禁止改写与禁止新增`
7. `补充背景`

这样做的目的是把“任务”和“事实”分开，把“关键约束”和“背景”分开。

---

## 7.2 Prompt 写法建议

### system 层
- 定义角色职责
- 定义约束优先级
- 定义禁止行为

### user 层
- 给出结构化章节目标
- 给出短句化事实块
- 给出输出格式要求

### 约束优先级建议
- `blockingConstraints > recentChanges > decisionContext > supportingContext`

### 负面规则建议
- 不要发明未被支持的重要事实
- 不要覆盖现有状态，除非章节目标明确要求推进
- 不要跳过关键因果过程

---

## 7.3 分阶段 Prompt 改造重点

### plan
- 重点要求模型输出“本章将改变什么、不改变什么”
- 让规划中显式映射召回事实，而不是只写剧情摘要

### draft
- 把 `blockingConstraints` 写成短句清单放前面
- 把“最近承接变化”单列，避免忘记承接最近状态

### review
- 改为缺陷导向，不再泛审美化
- 按“事实错 / 状态错 / 关系错 / 推进跳步 / 钩子遗漏”分类输出

### repair
- 要求只修关键缺陷，不主动扩写世界观

### approve diff
- 要求先判断是否发生真实事实变化，再输出结构化 diff

## 8. embedding / rerank 在 V3 中的定位

V3 不要求 embedding 或 rerank 立刻接管主召回，但建议把它们纳入正式计划。

### 当前状态

- 当前已生效的是**规则式召回 + 规则打分排序**
- 当前未生效的是**模型 rerank** 和 **embedding 召回**
- 因此 V3 的目标不是“把 rerank 和 embedding 从无到有硬塞进主链路”，而是先在现有规则排序基础上补齐实验链路和评测闭环

### embedding
- 定位为补充召回源
- 主要解决：同义表达、隐性关联、摘要化文本匹配不足
- 默认只补充 `decisionContext` 候选，不直接决定 `blockingConstraints`

### rerank
- 定位为候选精排层
- 先做启发式 rerank，再实验模型 rerank
- 主要服务于：降低噪声、提升 top N 质量、按阶段重排

### 原则
- 规则召回负责保底正确
- 规则 ranking 负责当前默认排序质量
- embedding 负责扩大语义覆盖
- rerank 负责提升排序质量

### 建议落地顺序

1. 先保留当前规则 ranking 作为基线
2. 增加 `HeuristicReranker`，验证是否能在不增加模型成本的情况下提升 top N 质量
3. 增加 `EmbeddingCandidateProvider`，只补充候选，不改写硬约束保底逻辑
4. 最后再实验模型型 rerank，只对 Top N 候选做二次精排

### 建议工程拆分

#### embedding 侧
- 新增 `EmbeddingProvider` 接口
- 新增实体摘要生成逻辑，而不是直接对原始长文本做 embedding
- 新增向量持久化表或可替换存储层
- 新增 `EmbeddingCandidateProvider`

#### rerank 侧
- 新增 `HeuristicReranker`
- 按阶段输入不同特征：`plan / draft / review`
- 后续再增加模型型 `RetrievalReranker` 实现

#### benchmark 侧
- 对比 `规则召回 + 规则 ranking`
- 对比 `规则召回 + heuristic rerank`
- 对比 `规则召回 + embedding 候选 + rerank`

## 8.1 embedding 字段选择原则

为了让后续 AI coding 更容易落地，embedding 相关设计建议遵守以下硬原则。

### 适合做 embedding 的字段

满足以下任一条件的字段，适合进入 embedding 摘要：

- 语义信息丰富
- 表达方式容易变化
- 用户意图和数据原文经常不是同一套词
- 关键词召回容易漏掉，但对剧情判断有帮助

典型包括：

- 人物动机
- 人物背景
- 势力目标
- 物品用途
- 关系描述
- 钩子铺垫与 payoff
- 世界规则解释
- 章节摘要

### 不适合做 embedding 的字段

以下字段不应交给 embedding 直接决定：

- 主键、外键、章节号
- 枚举状态字段
- 当前持有者、当前地点、当前所属势力
- 生死状态、关系类型、钩子状态等精确事实
- 数值边界、等级、金额、范围条件
- 必须严格遵守的禁止事项

这些字段更适合：

- 规则过滤
- 规则加权
- `blockingConstraints`
- `recentChanges`

### 一条实现原则

**embedding 只负责补充语义相关候选，不负责定义硬事实。**

---

## 8.2 embedding 输入不要直接用原始长文本

V3 不建议直接对数据库里的原始长文本做 embedding，而建议先生成“检索摘要”。

原因：

- 原始字段冗长，噪声高
- 多个字段混在一起时，模型不容易知道重点
- 相同实体在不同阶段需要的检索视角不同

建议每个实体先构造一段稳定、短小、可读的 embedding 摘要，再送去向量化。

建议摘要结构：

- `identity`
- `currentState`
- `coreConflictOrGoal`
- `recentChanges`
- `continuityRisk`

要求：

- 长度尽量稳定
- 优先短句
- 先保留最重要事实，再补充解释
- 不要把大量原始附注全文塞进去

---

## 8.3 各实体的 embedding 字段清单

以下清单按“适合 embedding / 不适合 embedding / 推荐摘要模板”的格式写，便于后续直接编码。

### characters

#### 适合 embedding
- `name`
- `alias`
- `goal`
- `background`
- `personality`
- `keywords`
- 角色摘要字段
- 最近章节中与该角色相关的行为摘要

#### 不适合 embedding
- `id`
- `book_id`
- 精确 `status`
- 精确 `current_location`
- 精确阵营归属
- 精确能力等级、货币数值、战力数值

#### 推荐摘要模板
```text
人物：<name>
别名：<alias>
身份摘要：<identity summary>
核心动机：<goal summary>
当前状态：<status + location + stance summary>
近期变化：<recent change summary>
连续性风险：<what must not be violated>
```

#### 适用场景
- 找与当前冲突语义相关的人物
- 找动机会驱动本章决策的人物
- 找“怀疑、隐忍、复仇、试探、合作但不信任”这类隐性心理线索

### factions

#### 适合 embedding
- `name`
- `category`
- `core_goal`
- 势力简介
- 势力当前战略状态摘要
- 与其他势力/人物的冲突摘要

#### 不适合 embedding
- `id`
- `book_id`
- 精确 `status`
- 精确控制区域标识
- 精确资源/等级/数值字段

#### 推荐摘要模板
```text
势力：<name>
类型：<category>
核心目标：<core goal>
当前态势：<current stance/pressure>
关键冲突：<major conflict>
近期变化：<recent change summary>
连续性风险：<what must not be violated>
```

#### 适用场景
- 找与当前场景、权力博弈、阵营冲突相关的势力
- 找“压制、拉拢、围剿、试探、渗透”这类语义关联

### items

#### 适合 embedding
- `name`
- `description`
- `ability`
- 用途摘要
- 与主线/身份/秘密相关的说明

#### 不适合 embedding
- `id`
- `book_id`
- 精确 `owner_type` / `owner_id`
- 精确 `status`
- 精确数量、稀有度枚举、消耗数值

#### 推荐摘要模板
```text
物品：<name>
用途：<usage summary>
关键能力：<ability summary>
当前状态：<owner/status short summary>
剧情关联：<why it matters>
近期变化：<recent transfer/change>
连续性风险：<what must not be violated>
```

#### 适用场景
- 找“关键证物、信物、身份凭证、限制器、禁器”这类变体表达
- 找没有直接写物品名、但语义上指向该物品的剧情需求

### relations

#### 适合 embedding
- 两端实体名称
- 关系描述摘要
- 当前张力摘要
- 最近关系变化摘要
- 冲突/互信/依赖说明

#### 不适合 embedding
- `id`
- `book_id`
- 精确 `relation_type`
- 精确 source/target id
- 精确状态枚举

#### 推荐摘要模板
```text
关系：<source> -> <target>
关系摘要：<relation summary>
当前张力：<current tension>
近期变化：<recent change>
风险提醒：<what progression must remain plausible>
```

#### 适用场景
- 找“松动、试探、暂时合作、表面敌对但互相利用”这类中间态关系
- 找与本章冲突推进最相关的关系线

### hooks

#### 适合 embedding
- `title`
- `foreshadowing`
- `expected_payoff`
- 钩子摘要
- 相关章节承接摘要

#### 不适合 embedding
- `id`
- `book_id`
- 精确 `status`
- 精确触发窗口编号
- 精确章节范围边界

#### 推荐摘要模板
```text
钩子：<title>
铺垫：<foreshadowing summary>
预期兑现：<expected payoff>
当前推进状态：<status short summary>
近期承接：<recent chapter linkage>
风险提醒：<too early / too late / easy to forget>
```

#### 适用场景
- 找“旧伏笔快要兑现”“该推进但还没回收”的剧情线
- 找语义上相关但标题没被直接提到的钩子

### world_settings

#### 适合 embedding
- `title`
- `category`
- 规则说明摘要
- 代价说明
- 禁忌说明
- 制度/体系摘要

#### 不适合 embedding
- `id`
- `book_id`
- 精确枚举状态
- 精确数值阈值
- 必须严格执行的硬边界原文

#### 推荐摘要模板
```text
设定：<title>
类别：<category>
规则摘要：<rule summary>
代价与限制：<cost/boundary>
禁忌：<forbidden moves>
连续性风险：<what must not disappear>
```

#### 适用场景
- 找“越级使用、透支、禁术、副作用、制度压力”这类规则边界
- 找用户没直接点名、但本章行为会触碰的世界规则

### chapters / plans / finals

#### 适合 embedding
- `summary`
- 章节事件摘要
- 结构化事实变更摘要
- 与当前章节相关的承接摘要

#### 不适合 embedding
- `id`
- `chapter_no`
- 精确状态字段
- 大段正文原文

#### 推荐摘要模板
```text
章节：第 <chapterNo> 章
核心事件：<event summary>
关键变化：<state changes>
承接价值：<why current chapter should care>
```

#### 适用场景
- 找最近承接关系
- 找类似情境和最近变更
- 给当前章节补足历史上下文

## 8.4 embedding 实现上的推荐拆分

为了便于 AI coding，建议把 embedding 逻辑按下面几个独立模块拆开。

### 模块一：摘要构造器

建议新增一层专门负责把数据库实体转换成 embedding 文本，例如：

- `buildCharacterEmbeddingText()`
- `buildFactionEmbeddingText()`
- `buildItemEmbeddingText()`
- `buildRelationEmbeddingText()`
- `buildHookEmbeddingText()`
- `buildWorldSettingEmbeddingText()`
- `buildChapterEmbeddingText()`

职责：

- 输入数据库记录
- 输出稳定的短文本摘要
- 不参与打分和检索

### 模块二：向量提供者

建议抽象：

- `EmbeddingProvider.embed(text)`
- `EmbeddingProvider.embedBatch(texts)`

职责：

- 把摘要文本转成向量
- 统一处理模型名、维度、失败重试

### 模块三：向量存储

建议单独抽象，不让 retrieval service 直接关心底层表结构。

职责：

- 保存实体摘要向量
- 按实体类型和模型版本刷新
- 查询近似候选

### 模块四：候选提供者

建议新增：

- `EmbeddingCandidateProvider`

职责：

- 根据 query 摘要拿到语义候选
- 输出与规则候选兼容的统一结构
- 不直接决定硬约束保底逻辑

### 模块五：合并与重排

建议让规则召回和 embedding 候选先 merge，再交给 reranker。

职责：

- 去重
- 保留 explainability
- 给出 `semanticScore`
- 与现有 `matchScore` / `importanceScore` 等特征并行存在

## 8.5 AI coding 友好的实现原则

为了让后续自动编码更可读、可维护，建议显式约束实现方式。

### 代码组织
- 每种实体一个摘要函数
- 每种实体一个字段映射常量
- 不要把所有 embedding 文本拼接逻辑塞进一个大函数

### 命名建议
- `build*EmbeddingText` 负责摘要构造
- `index*Embeddings` 负责建索引/刷新
- `search*EmbeddingCandidates` 负责查询候选

### 可读性要求
- 摘要模板优先短句，不要隐式拼接一长串字段
- 对“为什么这个字段能进 embedding”写简短注释
- 对“为什么这个字段不能进 embedding”在 plan 和 task 中保持一致表述

### 风险控制
- embedding 召回默认只进入 `decisionContext` 或 `supportingContext`
- 不允许 embedding 直接覆盖 `blockingConstraints`
- 精确事实仍以规则召回和结构化状态为准

## 8.8 最小 MVP 建议

为了降低首次落地复杂度，V3 不建议一上来覆盖所有实体。

### MVP 第一阶段

- 人物最容易产生动机、位置、人设错误
- 钩子最容易出现遗漏推进或兑现节奏错误
- 世界规则最容易在 draft 阶段被模型脑补覆盖

### MVP 第二阶段

再补：

1. `relations`
2. `items`
3. `factions`

## 8.9 推荐编码顺序

建议编码顺序如下。

1. 先补类型
- `types.ts` 或新增 `embedding-types.ts`
- `RetrievedFactPacket`
- `RetrievedPriorityContext`

2. 再补事实整理
- `retrieval-facts.ts`
- `retrieval-priorities.ts`
- `recent-changes.ts`

3. 再补 prompt 区块构造
- `prompt-context-blocks.ts`
- 让 `prompts.ts` 只负责模板拼装

4. 再补 embedding 摘要构造
- 先做 `characters/hooks/world_settings`
- 每种实体单独文件

5. 再补 provider 和 candidate merge
- `embedding-provider.ts`
- `embedding-candidate-provider.ts`

6. 最后补 heuristic rerank 与 benchmark
- `retrieval-reranker-heuristic.ts`
- `retrieval-benchmark.test.ts`

这个顺序的好处是：

- 前半段不依赖真实 embedding provider，也能先把结构理顺
- 可以先验证 prompt 质量提升，再接入向量能力
- 每一步都可以单独测试和回归

## 9. 建议实施顺序

### Phase 1. 建评测基线
- 固定案例集
- 标注关键事实
- 统计当前 V2 指标

### Phase 2. 做召回结构升级
- 增加事实包与最近变更视图
- 增加重要性分层

### Phase 3. 做 prompt 输入重组
- 把 JSON 大块替换成角色化上下文块
- 强化负面约束与优先级表达

### Phase 4. 做规则增强
- 加入意图槽位解析
- 加入 recent change / risk / forbidden rule 信号

### Phase 5. 做 rerank / embedding 实验
- 先 heuristic rerank
- 再 embedding candidate provider
- 最后做对照实验

## 10. 完成定义（DoD）

- 关键事实不会因 prompt 截断而频繁丢失
- 最近 1 到 3 章的关键状态变化可稳定进入上下文
- `plan / draft / review / approve diff` 的 prompt 均已使用阶段化事实块
- 高风险世界规则、关系状态、物品归属具备保底保留机制
- benchmark 已可稳定复跑，并能对比 V2 / V3 的效果
- embedding / rerank 的实验入口已纳入正式方案，但默认主链路仍可独立运行

---

## 11. 一句话结论

V3 的核心不是“召回更多”，而是：

**把最关键、最容易写错、最该影响当前章节决策的信息，稳定、简洁、按优先级地交给模型。**
